# Orchestrator Runbook — Provisioning a New Hydra Head

End-to-end procedure for using the orchestrator to spin up a fresh Hydra head, fund the admin wallet, and hand off to the middleware. Assumes a Linux host with Docker, `make`, and `jq`. Examples target preprod with the `lerna-labs/ekklesia-hydra` middleware; swap network and image for other scenarios.

## Prerequisites

1. **Cardano node for the target network is running and (close to) synced.**
   ```bash
   make NETWORK=preprod cardano-start
   make NETWORK=preprod cardano-status
   ```
   The hydra-node won't start cleanly if the node socket isn't responsive.

2. **Monitoring stack is up** (Prometheus + Grafana + node-exporter).
   ```bash
   make monitoring-start
   ```

3. **Env file layering is populated.** At minimum `.env` and `.preprod.env` with the usual keys (Blockfrost project id, TRP settings, etc.). Per-instance env files are generated by the orchestrator — don't create `.preprod.<instance>.env` by hand.

4. **Orchestrator API key is set.** Once per box:
   ```bash
   echo "ORCHESTRATOR_API_KEY=$(uuidgen)" >> .env
   ```
   Export it in the shell you'll use for curls:
   ```bash
   export ORCH_KEY=$(grep ORCHESTRATOR_API_KEY .env | cut -d= -f2)
   ```

## 1. Start the orchestrator

The orchestrator runs as a foreground Node process on the host (not in Docker — it needs `/proc` access and `make` invocation). Use tmux so you can detach without killing it:

```bash
tmux new -s orch
make orchestrator-start
# Ctrl+b then d to detach. Reattach later with: tmux attach -t orch
```

Alternative — background with a log file:
```bash
nohup make orchestrator-start > /tmp/orchestrator.log 2>&1 &
echo $! > /tmp/orchestrator.pid
tail -f /tmp/orchestrator.log
```

## 2. Confirm the orchestrator is healthy

```bash
curl -s localhost:7000/health | jq
```

Expect `canProvision: true`, `activeInstances: 0` (on a clean box), reasonable CPU/memory numbers. The `diskIopsRead`/`diskIopsWrite` values are cumulative counters from `/proc/diskstats` since boot — large numbers are expected, not a warning sign.

## 3. Pre-pull the middleware image (recommended)

Keeps the 120s readiness window from being consumed by a cold GHCR pull:

```bash
docker pull ghcr.io/lerna-labs/ekklesia-hydra:branch-main
```

If the image is private, `docker login ghcr.io` on this host first.

## 4. Scaffold the head

```bash
curl -X POST http://localhost:7000/heads \
  -H "Authorization: Bearer $ORCH_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "network": "preprod",
    "expressImage": "ghcr.io/lerna-labs/ekklesia-hydra:branch-main",
    "instanceName": "ekklesia-rss-1",
    "contestationPeriod": 600,
    "depositPeriod": 3600
  }'
```

Returns 202 with `status: "SCAFFOLDING"`. Field reference:

| Field | Required | Notes |
|-------|----------|-------|
| `network` | yes | `offline`, `preprod`, or `mainnet` (subject to `ALLOWED_NETWORKS`) |
| `expressImage` | yes | Middleware Docker image. Must match `IMAGE_ALLOWLIST_PATTERN` if set |
| `instanceName` | no | Lowercase alphanumeric + hyphens, 2-21 chars. Auto-generated if omitted |
| `contestationPeriod` | no | Close contestation window in seconds. Default 600. Use lower values (e.g. 120) only for dev speed — bump back up for real workloads |
| `depositPeriod` | no | Min time before deposit deadline. Default 3600 |

## 5. Poll until READY

```bash
watch -n 3 'curl -s localhost:7000/heads/ekklesia-rss-1 \
  -H "Authorization: Bearer $ORCH_KEY" | jq "{status, error}"'
```

Typical scaffold time is 30–60s. On `READY`, Ctrl+C out and pull the full payload:

```bash
curl -s localhost:7000/heads/ekklesia-rss-1 \
  -H "Authorization: Bearer $ORCH_KEY" | jq
```

Save three values for later:
- `adminAddress` — where you'll send funding
- `endpoints.express` — where the middleware is exposed
- `apiKey` — `x-api-key` header for middleware calls

## 6. Spot-check key ownership

Key generation shells out to Docker containers, which historically wrote files as `root:root` and caused `EACCES` when the orchestrator (running as the host user) tried to read them. The `user:` directive in `docker/docker-compose.keys.yml` + `HOST_UID`/`HOST_GID` exports in the Makefile fix this. Verify:

```bash
ls -la data/preprod/instances/ekklesia-rss-1/keys/
```

All files should be owned by your host user (e.g. `ubuntu ubuntu`), not `root root`. If you see root-owned files, the host-uid plumbing isn't taking effect — see **Troubleshooting**.

## 7. Fund the admin wallet on L1

Two things need to land at `adminAddress`:

**a. Base currency** — tADA on preprod from the Cardano faucet: https://docs.cardano.org/cardano-testnets/tools/faucet/. 10,000 tADA per request, confirms in ~30s. For a head that will see real traffic, hit the faucet twice so you've got commit fuel plus operational slack.

**b. Domain-specific tokens** — any vote tokens, governance NFTs, or other assets the middleware needs. Send from your test distribution wallet.

## 8. Verify funding landed

```bash
curl -s -H "project_id: $BLOCKFROST_PROJECT_ID" \
  https://cardano-preprod.blockfrost.io/api/v0/addresses/<adminAddress>/utxos | jq
```

Confirm both base currency and tokens are present before handing off to the middleware — an incomplete UTxO set will trip up `commit`.

## 9. Hand off to the middleware

The orchestrator's job ends here. Everything inside the head — commit, open, transact, decommit — is the middleware's responsibility. Exact routes depend on the middleware image; for `ekklesia-hydra` the open call is:

```bash
curl -X POST <endpoints.express>/start \
  -H "x-api-key: <apiKey>"
```

See the middleware's own docs for the full route surface.

## 10. Monitoring

- `docker ps | grep <instanceName>` — expect three containers (`hydra-node`, `hydra-trp`, middleware).
- `http://<host>:9090/targets` — Prometheus should discover the new target within ~15s via `file_sd_configs` (no restart needed).
- Host metrics dashboard in Grafana (`host-metrics.json`) — CPU / memory / disk / network for the orchestrator host.
- `tail -f data/orchestrator/audit.log` — JSONL trail of scaffold/teardown events.

## 11. Teardown

**Soft teardown** — stops containers, preserves env file + keys + registry entry for audit. Instance name is not reusable:

```bash
curl -X DELETE localhost:7000/heads/ekklesia-rss-1 \
  -H "Authorization: Bearer $ORCH_KEY"
```

**Full purge** — stops containers and wipes everything (env file, `data/{network}/instances/{id}/`, registry entry). Instance name becomes reusable:

```bash
curl -X DELETE "localhost:7000/heads/ekklesia-rss-1?purge=true" \
  -H "Authorization: Bearer $ORCH_KEY"
```

Use purge when an attempt failed and you want to retry with the same name, or when you're cleaning up a dev instance. Use soft teardown when you want the audit trail preserved.

## Troubleshooting

### Scaffold reports `FAILED` with `EACCES: permission denied` on a `.cardano.addr` file

The Docker key-gen containers wrote files as root, and the orchestrator (running as your host user) can't read them.

1. Confirm `docker/docker-compose.keys.yml` has `user: "${HOST_UID:-1000}:${HOST_GID:-1000}"` on the `x-keys` anchor.
2. Confirm the Makefile exports `HOST_UID` and `HOST_GID`:
   ```bash
   grep -n HOST_UID Makefile
   ```
3. **Restart the orchestrator after the fix** — its `process.env` is captured at startup, so changes to the Makefile don't reach existing instances.
4. Verify the env vars reached the running process:
   ```bash
   cat /proc/$(pgrep -f "tsx src/index.ts" | head -1)/environ | tr '\0' '\n' | grep HOST_
   ```
5. Clean up the root-owned leftovers before retrying:
   ```bash
   sudo chown -R $(id -u):$(id -g) data/preprod/
   rm -rf data/preprod/instances/<failed-instance>
   rm -f .preprod.<failed-instance>.env
   ```

### POST returns `Instance "X" already exists` after a prior attempt

Soft teardown and `FAILED` records remain in the registry. To reuse the name, either pick a new one or run a full purge:

```bash
curl -X DELETE "localhost:7000/heads/<instance-name>?purge=true" \
  -H "Authorization: Bearer $ORCH_KEY"
```

Purge also removes the env file and data directory, so the instance name is fully reusable on the next POST.

### Scaffold times out on readiness

Symptoms: `status: FAILED`, error mentions hydra-node readiness.

- Cardano node may not be synced enough. Check `make NETWORK=preprod cardano-status`.
- `READINESS_TIMEOUT_S` (default 120) can be too tight if the node was cold-started recently. Bump in `.env` and restart the orchestrator.
- Image pull may have eaten the window. Pre-pull the middleware image (step 3) before scaffolding.

### `docker ps` shows fewer than three containers for the instance

One of the containers crashed on startup. Inspect logs directly:
```bash
docker logs hydra-node-preprod-<instance>
docker logs hydra-trp-preprod-<instance>
docker logs express-preprod-<instance>
```

### Orchestrator process won't start

Usually a port conflict on 7000 or a missing `ORCHESTRATOR_API_KEY`. Check `.env`, or set `ORCHESTRATOR_PORT` to something free.
