# Hydra Notary

A document notarization example built on the Hydra SDK. This example demonstrates how to use a Hydra head as a fast, fee-less notarization layer — anchoring document hashes on-chain with full payloads pinned to IPFS for auditability.

## Overview

The notary service accepts documents, hashes them (SHA-256), pins the full payload to a local IPFS node, and records the IPFS CID alongside the document hash in Hydra transaction metadata. At settlement time, the entire collection of notarized documents can be published to IPFS as a single directory, producing one root CID that covers every document processed during the session.

### Architecture

```
Client --> Express API --> Disk (staging dir) --> IPFS Node (pin)
                |                                     |
                +--> Hydra Head (tx metadata: CID + doc hash)
                                    |
                            Settlement (L1)
                     single UTxO + IPFS root CID
```

### Key Concepts

- **IPFS CID Anchoring**: Rather than embedding variable-length document content directly in transaction metadata (which is constrained by Cardano's 64-byte metadata string limit), the full payload is pinned to IPFS. Only the fixed-size CID is stored on-chain.

- **Disk-Backed Cache**: Every notarization is written to disk before pinning to IPFS. A `latest/` directory maintains one file per document hash for fast lookups. On service restart, the in-memory cache rehydrates from this directory automatically.

- **Directory Publishing**: At settlement time, the entire `documents/` directory can be pinned to IPFS as a single directory object. The resulting root CID is a content-addressed commitment to every document notarized during the session — auditors can download the full set from one CID.

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/` | Health check |
| `GET` | `/health` | Hydra head status |
| `POST` | `/notarize` | Notarize a document (hash, pin to IPFS, anchor on-chain) |
| `GET` | `/document/:docHash` | Look up a notarized document by its SHA-256 hash |
| `GET` | `/documents` | List all notarized documents |
| `POST` | `/publish` | Pin the entire documents directory to IPFS and return the root CID |

### `POST /notarize`

```json
{
  "submitter": "addr_test1qz...",
  "filename": "contract.pdf",
  "content": "the document content"
}
```

**Response:**

```json
{
  "status": "SUCCESS",
  "data": {
    "docHash": "a1b2c3...",
    "ipfsCid": "QmXyz...",
    "timestamp": 1711700000000
  }
}
```

### `POST /publish`

No request body. Pins the full `documents/` directory to IPFS.

```json
{
  "status": "SUCCESS",
  "data": {
    "ipfsCid": "QmRootCid...",
    "size": 123456
  }
}
```

## Staging Directory Layout

Each Hydra instance gets its own isolated staging directory (mounted via Docker at `IPFS_STAGING_DIR`):

```
/ipfs-staging/
├── documents/          # Full payloads (pinned individually, and as a directory at publish)
│   ├── <hash>-<ts>.json
│   └── ...
└── latest/             # One file per doc hash (used for cache rehydration on restart)
    ├── <docHash>.json
    └── ...
```

## Prerequisites

- A running Hydra head (via `docker-compose.offline.yml` or a live network stack)
- A local IPFS node (Kubo) — added to `docker-compose.cardano.yml`
- The `IPFS_API_URL` and `IPFS_STAGING_DIR` environment variables set in your `.env` file

## tx3 Protocol

The `main.tx3` file defines a `notarize` transaction that:

1. Mints a `STAMP` token to the submitter as proof of notarization
2. Records the IPFS CID (label 0) and document hash (label 1) in transaction metadata

The protocol bindings are generated by running `trix bindgen` against `trix.toml`.
